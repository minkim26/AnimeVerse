# Supabase Setup

AnimeVerse uses Supabase for **Storage only** — not Auth, not the primary database. You need a Supabase project with two public Storage buckets: `avatars` (original uploads) and `avatar-thumbnails` (128x128 thumbnails generated by `consumer.ts`).

Use a project dedicated to AnimeVerse — don't reuse a Supabase project from another app, since the service_role key grants full access to everything in that project.

## 1. Create the project

1. Go to [supabase.com/dashboard](https://supabase.com/dashboard) and create a new organization/project if you don't already have one to use.
2. Click **New Project**, give it a name (e.g. `animeverse`), choose a region close to where you'll run the API, and set a database password.
   - AnimeVerse doesn't use Supabase's Postgres as its primary database (Prisma talks to its own `POSTGRES_URL`, typically the Docker Compose `postgres` service), so the generated database password only matters if you intend to point `POSTGRES_URL` at this Supabase project's Postgres instance for a hosted/testing environment. For pure local dev, you can otherwise ignore it.
3. Wait for provisioning to finish (a couple of minutes).

## 2. Create the Storage buckets

In the dashboard, go to **Storage** and create two buckets:

| Bucket | Public | Used for |
|---|---|---|
| `avatars` | Yes | Original uploaded avatar images (`api/avatar.ts`) |
| `avatar-thumbnails` | Yes | 128x128 thumbnails generated by `consumer.ts` |

Both need **public read access** so the frontend can render `avatarUrl` / `avatarThumbnailUrl` directly as `<img src>` without going through the API. When creating each bucket, toggle "Public bucket" on. The Express API and consumer write to these buckets using the service_role key (step 3), which bypasses any bucket-level RLS policies — public reads are what matters for the frontend.

## 3. Get your API credentials

In **Project Settings → API**, copy:

- **Project URL** → `SUPABASE_URL`
- **service_role key** (not the `anon` key) → `SUPABASE_KEY`

The service_role key has full read/write access to the project and bypasses Row Level Security. It is used exclusively server-side (`anime-verse-backend/lib/supabase.ts`) and must never be sent to the frontend or committed to git. Put both values in `anime-verse-backend/.env.local` (local dev) or `.env.production` (Docker Compose) — both files are gitignored.

```
SUPABASE_URL=https://<project-ref>.supabase.co
SUPABASE_KEY=<service_role key>
```

## 4. (Optional) Managing the project via the Management API

Everything above can also be done via [Supabase's Management API](https://api.supabase.com) using a personal access token (**Account → Access Tokens** in the dashboard) instead of clicking through the UI — useful for scripting project creation or bucket setup. If you go this route, treat the access token itself as a secret at least as sensitive as the service_role key: it can create and delete entire projects.

## Verifying the setup

Once `SUPABASE_URL` and `SUPABASE_KEY` are set and the backend is running (`docker compose up` or `npm run dev`), upload an avatar through the Profile page (or `POST /avatar` directly) and confirm:

1. The response includes an `avatarUrl` pointing at the `avatars` bucket.
2. After a few seconds, `GET /users/me` returns an `avatarThumbnailUrl` pointing at the `avatar-thumbnails` bucket (confirms the RabbitMQ consumer picked up the job and `sharp` thumbnailing succeeded).
3. Both URLs are publicly fetchable without any auth header — confirms the buckets are actually public.
